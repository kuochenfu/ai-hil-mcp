#### Watchpoint implementation note

probe-rs 0.31.0 has no high-level watchpoint API. Watchpoints are implemented by directly
writing to memory-mapped DWT registers:

```
DEMCR (0xE000EDFC)    ‚Äî bit 24 TRCENA must be set before DWT is accessible
DWT_CTRL (0xE0001000) ‚Äî bits [31:28] = NUMCOMP (number of comparators)
DWT_COMPn (0xE0001020 + n*0x10) ‚Äî comparator address
DWT_MASKn (0xE0001024 + n*0x10) ‚Äî 0 = exact address match
DWT_FUNCn (0xE0001028 + n*0x10) ‚Äî 0b0101=read, 0b0110=write, 0b0111=read_write
```

`set_watchpoint` enables TRCENA, reads NUMCOMP live from DWT_CTRL, scans for a free
comparator (FUNCTION[3:0] == 0), then writes COMP/MASK/FUNCTION.
`clear_watchpoint` scans all comparators for the matching address and zeros FUNCTION.

STM32WL55 CM4 has 4 DWT comparator units.

#### write_memory test note

First attempt at `0x20003F00` returned stale data on readback ‚Äî that address is inside the
active stack region (SP=`0x20003EC0`, stack grows down). Firmware overwrote the bytes during
the brief window between write and read. Second attempt at `0x20001000` (mid-SRAM, between
BSS end and stack) persisted correctly.

#### clear_breakpoint bug + real root cause

Initial fix (add `core.halt()` + `core.run()` around `clear_hw_breakpoint`) was not sufficient.
Root cause: probe-rs clears all FPB comparator registers on session open (or session drop).
Since each tool call opens its own session, breakpoints set in one call are gone by the next.

Additional finding: `clear_hw_breakpoint` in a new session cannot find the breakpoint because
probe-rs has no cross-session state. Fix replaced with raw FPB register scan:
- Read `FP_CTRL[7:4]` for `NUM_CODE` (number of comparators)
- Scan `FP_COMPn` registers for matching address with ENABLE=1
- Zero the matching comparator to clear

This mirrors the DWT watchpoint implementation (also raw register access).

### Hardware Test ‚Äî Session 3 (watchpoint halt + clear_breakpoint root cause)

| Tool | Result |
|------|--------|

| FPB persistence across sessions | ‚ùå FPB comparators cleared on every session open ‚Äî architectural limitation |
| **Watchpoint halt** | ‚úÖ CPU halted in `HAL_IncTick` at `str r3,[r2,#0]` writing `uwTick` (`0x200000DC`) |
| `clear_breakpoint` (raw FPB scan) | ‚úÖ Correctly returns "may already be cleared" when FPB is empty |

#### Watchpoint halt confirmation

Target: `uwTick` at `0x200000DC` ‚Äî written every 1ms by SysTick ISR.

After `set_watchpoint` + `resume_cpu`, `read_registers` caught CPU in watchpoint-halt state:
- `xPSR = 0xA100000F` ‚Üí exception #15 = **SysTick ISR** ‚Üê where the write happens
- `R2 = 0x200000DC` ‚Üê watchpoint address in register at halt point
- `R3 = 0x00044087` ‚Üê tick value being written
- `PC = 0x08002B32` ‚Üí epilogue of `HAL_IncTick`, one instruction after `str r3,[r2,#0]`

Disassembly confirms: DWT watchpoint fires on the `STR` instruction, CPU halts at the
following instruction. Halt occurs inside SysTick exception context, not thread mode.
