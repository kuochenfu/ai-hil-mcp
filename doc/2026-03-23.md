### `ppk2-mcp-rs` — Full Power Profiling MCP Server

New Rust MCP server wrapping the `ppk2 = "0.1.3"` crate over USB CDC-ACM.

#### Tools implemented (7 total)

| Tool | Description |
|------|-------------|
| `find_ppk2` | Auto-detect PPK2 control port — handles dual-port macOS enumeration |
| `measure_current` | Full current measurement: min/max/avg/std-dev/p50/p95/p99/energy/spike detection |
| `profile_power_states` | Histogram across log-scale current bands — identifies sleep/active/TX states |
| `measure_with_pin_trigger` | Current stats filtered to when a logic input pin (0–7) is HIGH or LOW |
| `estimate_battery_life` | Runs measurement + computes runtime hours/days given battery mAh |
| `set_dut_power` | Enable/disable DUT power without measuring |
| `get_metadata` | Read calibration status, VDD, HW version from device |

#### Architecture

Follows exact patterns from `jtag-mcp-rs` and `serial-mcp-rs`:
- Stateless per-call (each tool opens and closes the PPK2 connection)
- `do_*` synchronous implementation functions returning `Result<String>`
- Error caught at tool handler and returned as `"ERROR: ..."` string
- Semantic output — human-readable conclusions, not raw bytes

#### Key API decisions

- **`measure_current` output**: min/max/avg + std dev + p50/p95/p99 + energy in µJ and mWh + spike warning
- **`profile_power_states` buckets**: `< 1 µA` / `1–10 µA` / `10–100 µA` / `100µA–1mA` / `1–10mA` / `> 10mA` (log-scale decades matching typical embedded power modes)
- **`measure_with_pin_trigger`**: uses `start_measurement_matching(pins, sps)` — PPK2 hardware filters samples by logic pin state; `MeasurementMatch::NoMatch` counted for % time calculation
- **Spike threshold**: peak > 10× average → prepend `WARNING: CURRENT SPIKE DETECTED`
- **Measurement SPS**: 1000 (1ms resolution, good balance of detail vs. overhead)

### Hardware Issue: Dual CDC Port on macOS

**Problem**: `try_find_ppk2_port()` from the `ppk2` crate picks the first VID/PID match.
PPK2 enumerates two CDC-ACM ports on macOS:
```
/dev/cu.usbmodemF4C7372644342  ← control port (correct)
/dev/cu.usbmodemF4C7372644344  ← data/log port (wrong)
```
`find_ppk2` was returning `...344` (wrong) → `Ppk2::new()` → `Parse error in "Metadata"`.

**Fix**: Replaced `try_find_ppk2_port()` with custom port scan: collect all VID=0x1915/PID=0xc00a matches, sort lexicographically, return the lowest (`...342`). Report both ports to user.

**Also added**: 3-retry with 600ms delay around `Ppk2::new()` — device sometimes needs settling time.

### Hardware Test Results

**Setup**: STM32WL55 (NUCLEO-WL55JC1), JP1 jumper removed, powered exclusively via PPK2 source meter @ 3300 mV.

```
measure_current(port=".../342", mode="source_meter", voltage_mv=3300, duration_s=3)

Measurement complete (3.0s, 2996 samples)
Min:    -1.5 µA
Max:    22281.2 µA
Avg:    11005.5 µA
Energy: 108953.98 µJ
Peak/Avg ratio: 2.0x
```

**Interpretation**:
- ~11 mA average is consistent with STM32WL55 in active LoRa PING/PONG loop
- Peak 22 mA is plausible for SX126x PA ramp during TX
- Peak/Avg 2.0x — no spike warning (below 10× threshold)
- Negative min (-1.5 µA) is within noise floor — expected at this current range

**JP1 note**: JP1 isolates the MCU VDD from the ST-Link 3.3V rail on the NUCLEO board. Must be removed to make PPK2 the sole power source. Without removal, PPK2 fights the onboard regulator and readings are invalid.
