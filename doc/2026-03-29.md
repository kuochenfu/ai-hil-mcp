# 2026-03-29 — serial-mcp v2.0: multi-board support

## Summary

Enhanced `serial-mcp-rs` from a single-port tool to a full multi-board serial layer. Added a device registry (`devices.toml`), named board aliases, concurrent multi-port reads, and pattern-based event waiting. Tested live on 2 Espressif boards with 4 serial ports.

## Hardware

- **Board 1:** Espressif (USB JTAG `/dev/cu.usbmodem11201`, CP2102N UART `/dev/cu.usbserial-1110`)
- **Board 2:** Espressif (USB JTAG `/dev/cu.usbmodem11401`, CP2102N UART `/dev/cu.usbserial-1130`)
- Both boards running `synapse_s3gw` firmware with Zenoh transport

## Changes

### New: `devices.toml` device registry

Added a TOML config file loaded at server startup via `SERIAL_MCP_CONFIG` env var (set in `.mcp.json`). Maps human-readable board names to port paths and baud rates. Changes to the file take effect immediately without rebuilding the binary.

```toml
[board.board1]
log_port   = "/dev/cu.usbmodem11201"
shell_port = "/dev/cu.usbserial-1110"
baud       = 115200
```

Alias format: `<board>/<channel>` — e.g. `board1/log`, `board2/shell`. All tools accept either a raw path or an alias.

Config resolution order:
1. `SERIAL_MCP_CONFIG` env var (set per-project in `.mcp.json`)
2. Fallback: `devices.toml` next to the binary (global default)

### New tools (3)

| Tool | Description |
|------|-------------|
| `list_boards` | Show all configured boards with their aliases and resolved port paths |
| `read_multi_log(ports, lines, timeout_s)` | Read all ports concurrently via `tokio::task::spawn_blocking`; output interleaved by wall-clock timestamp |
| `wait_for_pattern(ports, pattern, timeout_s)` | Poll ports sequentially with 1s windows; return on first match; supports `\|` as OR |

### Updated tools (2)

- `list_serial_ports` — now includes USB hardware serial numbers (`[serial: ...]`) for stable board identification across reboots
- `read_serial_log` — added `timestamps: bool` parameter; prefixes each line with `[HH:MM:SS.mmm]`
- Both `read_serial_log` and `send_serial_command` now accept aliases

### Dependency added

- `toml = "0.8"` — config file parsing

## Implementation Notes

### Concurrent read architecture

`read_multi_log` spawns one `tokio::task::spawn_blocking` task per port (serial I/O is synchronous). Each task calls the shared `do_read_serial_log_raw()` function which returns `Vec<(SystemTime, String)>`. The main task joins all handles, merges, sorts by timestamp, and formats with `[<label>  HH:MM:SS.mmm]` prefix. Anomaly detection runs on the merged plain-text lines before output.

### Schema compatibility fix

Initial implementation used `Vec<String>` for the `ports` parameter in `read_multi_log` and `wait_for_pattern`. The rmcp/schemars schema generation for `Vec<T>` caused the MCP client to serialize the array as a JSON string literal rather than a JSON array, producing deserialization errors. Fixed by:
- Changing `ports: Vec<String>` → `ports: String` (comma-separated, split at tool boundary)
- Adding a `de_u64_or_str` custom deserializer to accept numeric params passed as strings

### Timestamps without extra deps

Wall-clock timestamps formatted using `std::time::SystemTime` + manual arithmetic (hours/minutes/seconds/ms from UNIX epoch). Avoids adding `chrono` as a dependency.

# 2026-03-29 (continued) — jtag-mcp v2.0 + build-flash-mcp v2.0

## Summary

Completed the multi-board upgrade across all three Rust MCP servers. jtag-mcp v2.0 adds probe selection by USB serial number, optional `board` param on all tools, `list_probes`/`list_boards` tools, and architecture-aware guards (Cortex-M vs ESP32/RISC-V). build-flash-mcp v2.0 adds a project registry, `board` param on all tools, flash tool abstraction (openocd/esptool/idf/probe-rs), architecture-aware size tool, and ESP-IDF support.

## jtag-mcp v2.0

### New features
- `list_probes` — enumerate all connected debug probes with serial numbers
- `list_boards` — show boards from `devices.toml` with their probe serials and targets
- Optional `board` param on all 8 existing tools — resolves probe serial + target from registry
- `is_cortex_m(target)` guard — returns user-friendly guidance instead of cryptic errors on ESP32/RISC-V targets for Cortex-M-only tools (`diagnose_hardfault`, `read_call_stack`)
- `open_session(probe_serial, target)` — selects specific probe by serial when `probe_serial` is `Some`
- `default_target` from `JTAG_MCP_TARGET` env var (fallback: `STM32WL55JCIx`)

### Key fix
- `resume_cpu` on ESP32-S3 returns "ERROR: An Xtensa specific error occurred" (probe-rs quirk). Use `reset_target` instead — documented in jtag user manual.

### New docs
- `doc/user-manual-jtag-mcp.md` — full reference manual, 14 tools, architecture awareness section, multi-board SOP

## build-flash-mcp v2.0

### New features
- `list_projects` — show boards configured in `devices.toml` with project paths, presets, flash tools
- Optional `board` param on all 4 tools — resolves `project_path`, `preset`, flash config from registry
- `BUILD_FLASH_MCP_CONFIG` env var in `.mcp.json`
- Flash tool abstraction:
  - `openocd` — dynamic scripts path via `brew --prefix open-ocd`; uses `-s <scripts_dir>` flag
  - `esptool` — parses `flasher_args.json` from ESP-IDF build, constructs correct write_flash command
  - `idf` — `source <idf_path>/export.sh && idf.py flash` via bash
  - `probe-rs` — `probe-rs download --chip <target>` with optional probe serial
- Architecture-aware size tool: `xtensa-esp32s3-elf-size` for ESP32-S3/S2, `riscv32-esp-elf-size` for ESP32-C3, `arm-none-eabi-size` for ARM

### ESP-IDF support (added during testing)
- `build_tool = "idf"` field — runs `bash -c "source <idf_path>/export.sh 2>/dev/null && idf.py -B <build_dir> build"`
- `idf_path` field — path to ESP-IDF root (project-local or system)
- `preset = ""` — tells server to use flat `build/` directory (ESP-IDF standard, no `build/Debug` subdirectory)
- Both board1 (synapse-s3gw) and board2 (synapse-s3node) configured; ESP-IDF at `/Users/chenfu/Labs/esp_projects/synapse-s3/esp-idf`
- `flasher_args.json` path resolves correctly with `preset = ""`

### New Cargo.toml deps
- `toml = "0.8"` — registry parsing
- `serde_json = "1"` — parse `flasher_args.json`
