# Dev Log — 2026-03-21

## Goals
- Port all MCP servers from Python to Rust for deployment simplicity (single static binary, no venv)
- Hardware-verify all Rust implementations against STM32WL55

## Completed

### Rust Track — All 3 MCP Servers Ported

#### `jtag-mcp-rs` (`probe-rs` + `rmcp`)
- Tools: `halt_cpu`, `resume_cpu`, `read_registers`, `read_memory`, `read_call_stack`, `diagnose_hardfault`
- Used `probe-rs` (Rust-native) instead of `pyocd` — more capable for STM32/nRF targets
- Binary size: ~10MB release
- **Hardware tested** on STM32WL55:
  - `halt_cpu` → PC = `0x08004102`
  - `read_registers` → all regs clean, Thread mode, SP in SRAM (`0x20003F8C`)
  - `read_memory(0x20000000, 32)` → valid SRAM data returned
  - `resume_cpu` → board running normally

#### `serial-mcp-rs` (`serialport` + `rmcp`)
- Tools: `list_serial_ports`, `read_serial_log`, `send_serial_command`
- Anomaly detection (HardFault, Panic, Watchdog) ported from Python
- **Hardware tested** on STM32WL55:
  - `list_serial_ports` → detected STLINK-V3 at `/dev/cu.usbmodem1303`
  - `read_serial_log` → captured live LoRa PING traffic (RSSI=-87/−88, SNR=12), no anomalies

#### `build-flash-mcp-rs` (`std::process::Command` + `rmcp`)
- Tools: `build_firmware`, `clean_build`, `get_build_size`, `flash_firmware`
- Supports CMakePresets.json and manual toolchain file, same as Python version
- **Hardware tested** full loop on STM32WL55 (`synapse-lora/CM4`):
  - `clean_build` → wiped `build/`
  - `build_firmware(preset="Debug")` → **SUCCESS** — `synapse-lora_CM4.elf` text=36072 data=168 bss=3152
  - `flash_firmware` → **SUCCESS** — verified and reset via ST-Link V3
  - `read_serial_log` → board alive, LoRa PING traffic confirmed

### MCP Config Updated
- `.mcp.json`: all 3 servers switched to Rust binaries
- Python versions kept as `*-py` fallback entries (`jtag-mcp-py`, `serial-mcp-py`, `build-flash-mcp-py`)
- `setup.sh`: auto-builds Rust binaries on first run; falls back to Python if `cargo` not available

### Toolchain Fix
- Root cause: Homebrew `arm-none-eabi-gcc` 15.2.0 no longer bundles newlib headers
- Fix: pinned `gcc-arm-none-eabi.cmake` to explicit ARM GNU Toolchain path at `/Applications/ArmGNUToolchain/15.2.rel1/arm-none-eabi/bin/`

## Decisions
- **Rust selected for deployment simplicity** — single static binary, zero runtime dependencies (no venv/pip). Performance was secondary; I/O-bound tools don't benefit meaningfully from Rust's speed.
- **Python fallback kept** — `vision-mcp` (future) stays Python due to OpenCV binding complexity. All Python servers remain in `.mcp.json` as `*-py` entries.
- **`probe-rs` over `pyocd`** — better STM32/nRF support, actively maintained, first-class Rust citizen.
